## 上下文

当前项目 README.md 为中文主文档，已覆盖产品背景和基础启动说明，但缺少面向国际读者的英文入口。用户希望将 README 重构为双语体系：英文作为主入口，中文作为本地化补充，并以链接方式互相跳转。该变更以 docs/design.md 的产品定义为事实来源，需保证文档内容与产品定位、页面结构、技术栈、路线图一致。

约束条件：
- 不修改运行时代码与构建配置。
- 保持现有仓库结构简洁，仅新增必要文档文件。
- 主入口必须是英文 README（README.md）。

## 目标 / 非目标

**目标：**
- 提供英文主 README，面向外部开发者快速理解项目价值与使用方式。
- 提供中文 README，保留本地读者可读性与完整信息。
- 在英文 README 中显式链接到中文 README，并在中文 README 回链英文 README。
- 对齐 docs/design.md 中的产品信息，避免描述漂移。

**非目标：**
- 不新增产品功能或页面交互。
- 不调整 API、依赖、打包策略。
- 不将 PRD 全量原文搬运到 README（避免冗长）。

## 决策

1. 文件命名策略
- 决策：保留 README.md 作为英文主入口，新增 README.zh-CN.md 作为中文文档。
- 理由：README.md 是 GitHub 默认入口，英文更利于项目引流与传播；zh-CN 后缀语义清晰。
- 替代方案：README.en.md + README.zh.md。
- 未采用原因：会影响默认入口体验，增加访问一步跳转。

2. 内容结构策略
- 决策：两份文档使用相同章节骨架（Overview、Core Features、Architecture、Getting Started、Pages、Roadmap）。
- 理由：降低双语维护成本，避免信息不对称。
- 替代方案：中文长文、英文短文。
- 未采用原因：会导致版本演进时内容漂移。

3. 信息来源策略
- 决策：以 docs/design.md 为功能与术语基准，结合现有 README 的工程落地信息。
- 理由：design 文档给出产品边界，现有 README 包含已落地工程信息，两者结合可保证准确与可执行。
- 替代方案：仅基于现有 README 重写。
- 未采用原因：可能遗漏最新 PRD 约束（如坐标系统原则）。

4. 变更范围策略
- 决策：仅修改文档文件，不触及 src/ 代码。
- 理由：该需求属于文档升级，应保持实现风险最低。
- 替代方案：同步添加应用内帮助页。
- 未采用原因：超出本次目标与发布窗口。

## 风险 / 权衡

- [双语内容后续不一致] → 建议在 README 中保持章节一一对应，并在后续变更中同步更新两份文档。
- [信息过载导致转化下降] → 英文主 README 控制篇幅，保留高价值信息并链接更详细文档。
- [术语翻译不统一] → 固定关键术语映射（例如 World Coordinate、Viewport、Hotspot）。

## 迁移计划

1. 重写 README.md 为英文主文档并加入中文入口链接。
2. 新增 README.zh-CN.md，按同结构提供中文内容并回链英文文档。
3. 自检两份文档的章节对齐、命令可执行性与链接可访问性。
4. 提交后由评审确认文案准确性与信息完整性。

回滚策略：若出现文案争议，可回退到旧版 README.md，并保留本次变更分支继续迭代。

- 是否需要在 README 中补充截图或 GIF 以进一步提升引流效果（当前变更先聚焦文本结构）。
