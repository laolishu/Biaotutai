## 上下文

当前主题状态管理已通过 `useTheme` 实现，并且主题 token 已覆盖 Upload、Workbench、Viewer 三个页面，但主题切换入口主要集中在 Workbench。用户从 Upload 进入 Viewer 的流程中缺少就地切换入口，导致可操作性与体验一致性不足。

本次变更仅扩展入口，不调整主题状态来源、持久化键和 token 机制。

## 目标 / 非目标

**目标：**
- 在 Upload 与 Viewer 页面增加主题切换入口
- 与现有切换行为保持一致（切换 dark/light，同步 html `data-theme` 与 localStorage）
- 保持按钮语义与可访问性一致（title、aria-label）

**非目标：**
- 不重构 `useTheme` 逻辑
- 不新增第三种主题模式
- 不修改主题 token 定义与命名

## 决策

### 决策 1：复用现有 `useTheme`，不新增页面级状态

- 原因：主题状态属于全局状态，入口应复用而非分叉
- 备选方案：在 Upload/Viewer 各自维护主题状态（否决，容易与全局状态不一致）

### 决策 2：Upload 与 Viewer 使用一致的图标按钮样式和语义结构

- 原因：降低用户学习成本，统一跨页面交互语言
- 备选方案：两个页面使用不同样式入口（否决，会造成风格割裂）

### 决策 3：入口位置采用页面顶部右上区域

- 原因：符合用户对主题控制作为“全局显示偏好”的预期位置
- 备选方案：放入页面内容区（否决，可发现性较差）

## 风险 / 权衡

- [页面布局冲突] 新增按钮可能影响原有顶部排版 → 通过最小侵入布局调整与统一按钮尺寸规避
- [可访问性遗漏] 仅放图标可能语义不足 → 强制要求 title 与 aria-label
- [行为偏差] 不同页面绑定不同事件逻辑 → 统一调用 `toggleTheme()` 并复用同一条件渲染

## 迁移计划

1. 在 Upload 页面接入主题切换按钮并绑定 `toggleTheme()`
2. 在 Viewer 页面接入主题切换按钮并绑定 `toggleTheme()`
3. 校验 dark/light 切换、刷新后持久化恢复、语义属性完整性

## 待确认问题

- Upload 与 Viewer 是否需要与 Workbench 完全一致的图标形态（太阳/月亮）而非文本按钮
- 入口在移动端窄屏时是否需要降级布局策略（例如仅图标不显示旁文）
