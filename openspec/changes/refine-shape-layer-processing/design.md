## 上下文

当前 `shape` 图层在解析与标准化流程中缺乏细粒度语义，导出 JSON 面向对接方时只能表达“形状存在”，难以稳定映射到可编辑结构组件（如矩形卡片）。
同时，复杂路径图层（不规则 path / 多子路径组合）如果强行结构化，容易造成导入端渲染偏差。项目已具备 `assets` 外联机制，因此可利用“复杂图层图片降级”保证视觉一致性。

## 目标 / 非目标

**目标：**
- 为 `shape` 图层增加可判定的细化路径：`结构组件识别`、`样式保留`、`复杂路径降级`。
- 在导出 JSON 中稳定输出 `fill`、`cornerRadius`、`stroke` 等属性，供导入方直接消费。
- 对复杂 path 提供统一降级协议（导出图片 + `assetId` 关联 + 降级标记）。
- 保持与现有导出规范兼容，不破坏已有 image/text/group 解析链路。

**非目标：**
- 不在本次变更中实现运行时交互逻辑（绑定、事件、动画）。
- 不覆盖所有矢量路径布尔运算语义，仅提供“可结构化/不可结构化”分层处理。
- 不引入新的渲染引擎或替换现有坐标系统。

## 决策

### 决策 1：`shape` 优先进行结构组件识别
- 方案：当图层满足规则（矩形边界明确、单一填充/描边、可读圆角）时识别为结构组件，类型标记为 `shape`，并在 `props.shapeType` 写入 `rect` 或 `card`。
- 理由：便于导入端直接创建可编辑节点，而非二次几何推断。
- 备选方案：统一降级为图片。该方案实现简单，但失去编辑能力，因此不采用。

### 决策 2：保留视觉样式为规范字段
- 方案：标准化输出 `props.fill`、`props.stroke`、`props.cornerRadius`、`props.opacity`（有值时）。
- 理由：这些字段是导入端还原形状最小必要集合，且与当前文档约定一致。
- 备选方案：将样式打包为未结构化 `style` blob。该方案可读性差、对接成本高，不采用。

### 决策 3：复杂路径统一降级为图片
- 方案：对无法稳定结构化的复杂路径，导出切图到 `assets`，layer 改为 `type=image` 或在 `shape` 上标记 `props.degraded=true` + `props.assetId`（以实现方案为准）。
- 理由：保证视觉保真优先，避免导入端出现错误几何。
- 备选方案：导出 path 命令。该方案实现复杂，跨端兼容风险高，MVP 阶段不采用。

### 决策 4：在属性面板与规范中暴露降级状态
- 方案：对被降级层输出可机读标记（如 `props.degradeReason`），并在图层信息中可见。
- 理由：帮助对接方和开发者快速定位“为何不是结构组件”。
- 备选方案：仅日志输出。日志不可随数据流传递，不利于排障。

## 风险 / 权衡

- [风险] 结构识别规则过严导致可编辑层过少 → [缓解] 逐步放宽规则并在规范中明确判定条件。
- [风险] 规则过松导致错误识别（视觉偏差）→ [缓解] 优先保证保真，识别不确定时强制降级图片。
- [风险] 图片降级增加资源体积 → [缓解] 复用 `assets` 去重与 hash 校验，后续可增加压缩策略。
- [权衡] 选择“保真优先”会牺牲部分可编辑性，这是面向导入稳定性的有意识取舍。

## Migration Plan

1. 在 shape 标准化流程中新增判定与分流逻辑（结构化 / 降级）。
2. 更新导出 JSON 文档与 capability 规范，定义新增字段与约束。
3. 对导入端提供兼容建议：优先消费结构化字段，检测降级标记后按图片渲染。
4. 回滚策略：若识别逻辑异常，可暂时将所有 shape 走降级图片路径，确保线上视觉稳定。

## Open Questions

- `card` 与 `rect` 的边界判定阈值（圆角占比、描边存在性）是否需要配置化？
- 降级后 layer 是否统一转成 `type=image`，还是保留 `type=shape` + `assetId` 双态？
- 对渐变填充的结构化表示是否在本次纳入（当前建议先支持纯色优先）。
