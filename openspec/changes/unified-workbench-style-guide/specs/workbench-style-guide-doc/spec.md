## 新增需求

### 需求:必须提供业务中立的工作台 UI 风格指南文档
系统必须在 `docs/workbench-style-guide.md` 提供一份业务中立的工作台 UI 布局与交互规范文档，覆盖五区布局模型、Activity Bar 交互模式、Activity Panel 状态、Canvas 区、Header 操作区、拖拽调宽、视觉 token 引用策略及多产品接入约定。

#### 场景:文档可被多产品团队引用
- **当** 一个新产品需要实现三栏式工作台布局
- **那么** 开发者必须能通过 `docs/workbench-style-guide.md` 理解布局结构、状态模型与交互规则，无需阅读业务相关源码

### 需求:风格指南必须使用业务中立术语
文档中所有模块命名必须与具体产品业务解耦，禁止在框架定义章节出现"图层""PSD""标注""大屏""组件库"等产品专属术语；必须使用"条目树""资源树""主视图""属性面板"等中性命名，并提供业务映射表说明各产品如何对应框架角色。

#### 场景:映射表覆盖至少两个产品
- **当** 读者查阅文档中的"多产品业务映射"章节
- **那么** 文档必须以表格形式列出至少两个产品（标图台、大屏设计器）在各区域的业务填充内容

### 需求:风格指南必须定义 Activity Bar 的交互协议
文档必须明确描述 Activity Bar 的三种状态转换规则：点击非激活图标展开对应面板、点击已激活图标收起面板、收起后再次点击恢复展开；该规则必须适用于左侧与右侧两套 Activity Bar，且描述方式与具体图标或业务无关。

#### 场景:左右 Activity Bar 遵循相同协议
- **当** 读者阅读 Activity Bar 交互协议章节
- **那么** 文档必须说明左右两侧均遵循同一套三态切换规则，无需分别描述

### 需求:风格指南必须定义拖拽调宽的交互规则
文档必须描述 Activity Panel 拖拽调宽的交互模型：仅展开状态下显示 Resize Handle、mousemove 驱动宽度变化（含 clamp 边界）、拖拽中临时禁用过渡动画；左右两侧的方向差异必须明确说明。

#### 场景:左右拖拽方向差异有明确说明
- **当** 读者实现右侧面板拖拽调宽
- **那么** 文档必须明确指出右侧 Resize Handle 位于面板左侧边缘，且宽度变化方向与左侧相反

### 需求:风格指南必须使用 CSS token 引用策略描述视觉规范
文档的视觉规范章节必须以 `--color-*` CSS 自定义属性 token 名称描述颜色语义，禁止写死具体色值；必须说明各产品通过配置 token 值来适配自身品牌风格，框架级视觉结构保持统一。

#### 场景:视觉规范描述 token 语义而非色值
- **当** 读者查阅视觉规范章节
- **那么** 文档必须列出 token 名称（如 `--color-bg-base`、`--color-accent`）及其语义说明，而非具体颜色值
