## 上下文

标图台已实现完整的三栏工作台布局，包含：左侧 Activity Bar + Activity Panel、中间 Canvas 区、右侧 Activity Panel + Activity Bar、顶部 Header。该实现模式适合所有以"资源树 + 画布 + 属性"为核心的创作类工具（PSD 标注、大屏设计、原型设计等）。

当前问题：设计知识只存在于代码实现中，无法跨产品传递；各产品若独立决策将产生交互漂移。

## 目标 / 非目标

**目标：**
- 将现有实现的布局模型、交互模式、视觉 token 策略以文档形式沉淀到 `docs/`
- 文档使用业务中立术语，使其对多个产品均可复用
- 定义工作台框架的"接入约定"，使接入产品只需填充业务内容

**非目标：**
- 不包含 PSD 解析、大屏组件库等业务逻辑
- 不是代码库或 UI 组件库
- 不要求现有实现立即修改

## 决策

### 文档放置位置：`docs/workbench-style-guide.md`

- 与现有 `docs/design.md` 和 `docs/ui-design/` 同层
- 面向设计/开发阅读，不需要构建工具支持
- 替代方案（放在 `openspec/specs/`）：否决，openspec specs 是变更增量文件，不是独立文档

### 文档语言：业务中立

- 避免"图层/PSD/标注"等产品专属术语
- 使用"条目树、资源树、主视图、属性面板"等中性命名
- 提供映射表说明各产品如何对应框架中的角色

### 视觉规范：引用 token 名称，不写死色值

- 文档描述 `--color-bg-base`、`--color-accent` 等 token 语义
- 各产品配置各自的 token 值，框架约定不变

## 风险 / 权衡

- [文档滞后于实现] → 文档以当前实现为基准，后续实现变更时需同步更新文档
- [各产品对框架的理解偏差] → 文档中"接入约定"章节明确扩展边界，减少歧义
